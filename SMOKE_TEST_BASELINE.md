# SMOKE TEST BASELINE

Дата фиксации baseline: 2026-04-24 (UTC).

Цель документа: зафиксировать текущее поведение CRM-прототипа перед любыми архитектурными изменениями и определить ключевые smoke-сценарии, которые **нельзя ломать**.

## Предусловия
- Проект работает в demo/mock-режиме на локальных mock-данных.
- Маршрутизация выполняется через `react-router`.
- Основные разделы доступны из sidebar.

## Ключевые сценарии, которые должны оставаться рабочими

### 1) Открытие dashboard
- Переход по маршруту `/dashboard` открывает страницу «Операционный обзор».
- Отображаются метрики и таблицы «Последние изменения по субъектам» / «Последние поручения».
- Ссылка «Все поручения» переводит в `/requests`.

### 2) Переходы по sidebar
- Пункты и группы sidebar открывают соответствующие маршруты без ошибок и пустых экранов:
  - `/dashboard`
  - `/subjects`
  - `/brokerage`
  - `/trust-management`
  - `/agents`
  - `/archives`
  - `/requests`
  - `/compliance`
  - `/middle-office`
  - `/back-office`
  - `/trading`
  - `/depository`
  - `/administration`

### 3) Список субъектов
- Переход на `/subjects` показывает таблицу активных субъектов (неархивные записи).
- Кнопка «+ Добавить» открывает `/subjects/register`.
- Клик по строке субъекта открывает карточку субъекта `/subjects/:id`.

### 4) Фильтры / поиск / сортировка / пагинация субъектов
- Поиск по значениям клиента (ФИО/название, код, ИНН, email, телефон).
- Фильтры по типу, резидентству, статусу субъекта, статусу комплаенса, квалификации, роли.
- Сортировка по ключевым колонкам (включая переключение asc/desc).
- Пагинация и выбор размера страницы работают корректно.
- Сброс фильтров возвращает полный список.
- Параметры фильтрации синхронизируются с query-параметрами URL.

### 5) Открытие карточки субъекта
- Маршрут `/subjects/:id` открывает карточку выбранного субъекта.
- Работают табы карточки (профиль, договоры, банковские счета, документы, связи, история).
- Доступно редактирование профиля без падений интерфейса.
- Доступна архивация субъекта и возврат в рабочие сценарии.

### 6) Регистрация физлица
- Маршрут `/subjects/register` поддерживает сценарий выбора «Физическое лицо».
- Валидация минимально обязательных полей отрабатывает.
- Успешное завершение создает клиента в mock-store и показывает результат регистрации.

### 7) Регистрация юрлица
- Маршрут `/subjects/register` поддерживает сценарий выбора «Юридическое лицо».
- Валидация обязательных полей отрабатывает.
- Успешное завершение создает клиента в mock-store и показывает результат регистрации.

### 8) Договоры / счета
- В карточке субъекта доступен таб договоров и связанные действия.
- В карточке субъекта доступен таб банковских счетов (просмотр/обновление/добавление).
- Wizard договора (`/subjects/:subjectId/contract-wizard`) открывается и завершает сценарий без ошибок UI.

### 9) Поручения
- Страница `/requests` открывает список поручений.
- Доступны базовые фильтры/поиск и операции создания поручений в рамках mock-сценария.

### 10) Комплаенс
- Страница `/compliance` открывает список клиентов комплаенса.
- Работают фильтры и открытие карточки `/compliance/:id`.
- Бейджи статусов/типов/резидентства отображаются корректно.

### 11) Трейдинг
- Страница `/trading` открывает список трейдинг-профилей.
- Работают поиск, фильтры, сортировка и пагинация.
- Открывается карточка `/trading/:id`.

### 12) Депозитарий
- Страница `/depository` открывает раздел отчетов депозитария.
- Работают фильтры (код, даты, тип отчета, канал, результат), выбор отчета, действия «повторить отправку/скачать» в mock-режиме.

### 13) Архивы
- Страница `/archives` показывает архивные субъекты.
- Работают поиск/фильтр по типу.
- Кнопка «Восстановить» возвращает субъекта в активный список.

### 14) Администрирование
- Страница `/administration` открывается и отображает карточки секций администрирования.

## Технический baseline сборки
- `npm install` выполнен успешно.
- `npm run build` выполнен успешно:
  - TypeScript компиляция прошла.
  - Production build Vite собран без ошибок.

## Границы изменений для следующих PR
- Не удалять существующие маршруты, страницы, mock-данные и пользовательские сценарии.
- Не подключать реальный backend, URL прод-сервисов и API-ключи.
- Любые рискованные изменения внедрять через совместимые обертки.
