# АУДИТ МИГРАЦИИ: Compliance — Список и Карточка
### UmiCMS → Новый статический CRM-дизайн

---

## 1. Краткое резюме

`content-116.phtml` является **страницей списка комплаенса**, которая совмещает: вывод клиентов со статусом 3 («На проверке»), все бизнес-действия (одобрить / на доработку / заблокировать) и inline-просмотр данных через модальные окна UIKit. Отдельной «карточки» клиента в старом коде **не существует**.

Новый дизайн разделяет функциональность на **две страницы**: реестр (`compliance.html`) и карточку проверки (`compliance-card.html`). Это архитектурное изменение является главным вызовом миграции: вся бизнес-логика из phtml переходит в `content_custom`, а inline-модалки заменяются переходом на отдельную страницу.

**Полностью подтверждены из старого кода:** status_klienta (значения 3/4/5/-1), hash, basicname, inn, num_investika, surname/namecl/patronymic, komplajns_zamechaniya, все 7 обработчиков действий, DualLog.

**Не найдены в старом коде:** резидентство, подтип ЮЛ, ИП-тип, KYC/AML статусы, документы, риск-уровень, контакты, офицер, дата обновления, черновик, экспорт CSV.

## 3. Что делает старый content-116.phtml

### 3.1. Является ли это страницей списка?

**Да.** Заголовок: «Клиенты для проверки». Две вкладки (switcher): «Юридические лица» / «Физические лица». Каждая вкладка — таблица клиентов.

### 3.2. Является ли это страницей карточки?

**Нет.** Карточки нет. Детали клиента открываются через inline UIKit-модалки прямо в списке.

### 3.3. Совмещает ли логику списка и карточки?

**Да.** Всё в одном файле: список + модалки с деталями + обработчики действий.

### 3.4. Содержит ли обработчики POST/GET?

**Да.** 7 обработчиков в строках 144–506. Все запускаются до HTML-вывода и завершаются `header("Location:") + exit`.

### 3.5. Является ли это только рендерингом или бизнес-логикой тоже?

**Оба.** Файл содержит полноценную бизнес-логику: изменение `status_klienta`, обновление `komplajns_zamechaniya`, DualLog. Это не просто шаблон.

### 3.6. Вспомогательные функции (строки 15–131)

| Функция | Назначение | Строки |
|---------|-----------|--------|
| `normForLog($v)` | Нормализация значения для лога | 15–24 |
| `trackUmiSet($object, &$fieldsLog, $field, $newValue)` | `setValue` с записью diff в лог | 26–64 |
| `getUserIdByStatus($status_klienta_input)` | `selector('objects')`, тип `ClientPhys`, фильтр по `status_klienta` | 67–73 |
| `getFirmsIdByStatus($status_klienta_input)` | `selector('objects')`, тип `ClientFirm`, фильтр по `status_klienta` | 75–81 |
| `addChatLines($date)` | Рендеринг истории комментариев из массива строк (UIKit panel) | 83–108 |
| `safeValue($val)` | `htmlspecialchars` для значений любого типа (null/scalar/array/object) | 113–130 |

## 15. Риски и вопросы к владельцу проекта

### Технические риски

**Р1. Два selector'а вместо одного.** UmiCMS selector работает с одним типом объекта. Объединение firm + phys требует двух запросов и PHP-merge. Глобальная сортировка и точный `offset` нетривиальны.

**Р2. Тип данных `status_klienta`.** В старом коде: `4` (int, стр. 159), `"5"` (string, стр. 368), `"-1"` (string, стр. 262). Несогласованность типов может привести к ошибкам. Стандартизировать до int перед миграцией.

**Р3. `$legal_point_check_info_firms` / `$legal_point_check_info_phys` не определены.** Используются в строках 617, 727, но нигде в файле не объявлены. Источник (вероятно `initialize.php` или другой include) неизвестен. Без них блок «Подробнее» пуст.

**Р4. Старые GET-обработчики нельзя удалять досрочно.** `?checked=`, `?checked_phys=`, `?redact_phys=` — прямые ссылки в HTML. До полного переключения на новый UI они должны работать.

**Р5. Большое количество полей карточки без источника.** Около 60% полей карточки не найдены в `content-116.phtml`. Начинать реализацию карточки без инвентаризации UMI-полей — риск переделки.

**Р6. Черновик и CSV** — полностью новые функции без прецедента в старом коде.

### Вопросы к владельцу проекта

1. Полная схема полей объектов `ClientFirm` и `ClientPhys` в UmiCMS (список + типы)?
2. Что содержат `$legal_point_check_info_firms` и `$legal_point_check_info_phys`? В каком файле определяются?
3. Как определить ИП: отдельный `TmpltId::$ClientIP`? Поле `is_ip`? По ОГРНИП?
4. Существует ли поле `org_type` (ООО/АО/ПАО)? Какие значения принимает?
5. Существует ли поле `residency`? Какой тип и значения?
6. Как хранятся документы клиента: поля объекта или связанные UMI-объекты?
7. Как вычисляется «Полный комплект документов» — ручной флаг или автоматически?
