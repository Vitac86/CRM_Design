# Аудит UI-паритета: HTML реальной CRM и Static UIKit

## 1. Краткое резюме

**Общий уровень структурного паритета: средний.**

Снимок реальной CRM и Static UIKit используют общий набор CRM-компонентов: оболочку приложения, боковую панель, верхнюю панель, главный блок субъекта, карточки, вкладки, таблицы, кнопки и бейджи. При этом внутренняя разметка карточки субъекта различается в нескольких важных местах.

Основным объектом сравнения является `subject-card-individual.html`, поскольку в снимке реальной CRM источник субъекта указан как `physical_persons`. Страница организации `subject-card.html` используется только для дополнительной диагностики. Даже страница физического лица не является полностью идентичным примером: в ней отличаются демонстрационные данные, доступные действия, состояния и пользовательские сценарии.

Наиболее важные различия обнаружены в иерархии действий главного блока, порядке вкладок и начальном состоянии панелей, классах карточек профиля, схемах таблиц и отсутствии в Static UIKit модального окна закрытия договора.

Немедленные крупные исправления не рекомендуются. Сначала следует подтвердить, что именно страница физического лица является целевой для достижения паритета, а также собрать визуальные материалы. Позже можно безопасно выполнить несколько небольших структурных изменений, но их не следует объединять со спекулятивными правками CSS.

## 3. Основной вывод

**Есть ли различия?** Да. Обнаружены явные различия в DOM-структуре, классах, обёртках, вкладках, таблицах, действиях и пользовательских сценариях.

**Критичны ли они?** Некоторые различия могут существенно влиять на отображение или поведение. В первую очередь это вложенность действий главного блока, схемы таблиц, начальное состояние активной панели и модальное окно закрытия договора. Другие различия, вероятно, ожидаемы, поскольку страницы демонстрируют разные данные и состояния приложения.

**Является ли Static UIKit полностью неправильным?** Нет. В нём сохранена основная модель CRM-компонентов и многие важные имена классов. Оболочка, главный блок субъекта, вкладки, классы таблиц UIkit, бейджи и основные разделы карточки имеют узнаваемые аналоги в реальной CRM.

**Достаточно ли текущего снимка для окончательного визуального паритета?** Нет. По одному HTML невозможно подтвердить точный визуальный паритет. Необходимы CSS реальной CRM, вычисленные стили, снимки экрана, сведения о размере окна браузера и состояние DOM после выполнения JavaScript.

## 5. Подробные результаты

### 5.1 Корень страницы и оболочка

**Структура реальной CRM:** Основное содержимое непосредственно содержит `<section class="crm-page" data-page="subject-card">`, внутри которого находится общая оболочка карточки субъекта.

**Структура Static UIKit:** Основное содержимое содержит `<section id="crm-page">`, а внутри него — `<div class="crm-page" data-page="subject-card">`.

**Различие:** Static UIKit добавляет дополнительную обёртку и использует другой семантический элемент для класса `.crm-page`.

**Вероятное влияние:** Могут по-разному работать селекторы непосредственных дочерних элементов и правила компоновки. Фактическое визуальное влияние невозможно подтвердить только по HTML.

**Уверенность:** Высокая

**Нужны ли CSS или вычисленные стили?** Да

### 5.2 Боковая и верхняя панели

**Структура реальной CRM:** Дашборд является навигационным пунктом верхнего уровня. Фронт-офис и мидл-офис представлены отдельными группами с CRM-символами SVG. Верхняя панель содержит поиск, сведения о пользователе, роль и ссылку выхода.

**Структура Static UIKit:** Дашборд находится внутри подменю фронт-офиса. Отличаются некоторые фрагменты значков и подписи, а для мидл-офиса используется отдельное изображение. Верхняя панель сохраняет основные обёртки, но содержит упрощённые данные пользователя.

**Различие:** Иерархия навигации и сопоставление значков не идентичны; содержимое верхней панели эквивалентно лишь частично.

**Вероятное влияние:** Могут отличаться отступы и группировка боковой панели, поведение активных пунктов, вид значков и ширина пользовательского блока.

**Уверенность:** Высокая

**Нужны ли CSS или вычисленные стили?** Да

### 5.3 Главный блок субъекта

**Структура реальной CRM:** `crm-subject-actions` вложен в `crm-subject-badge-groups` внутри области идентификации. Главный блок содержит атрибуты с типом субъекта, источником и статусом комплаенса.

**Структура Static UIKit:** `crm-subject-actions` является непосредственным дочерним элементом главного блока и расположен отдельно от `crm-subject-hero-main`. Вид субъекта хранится в атрибуте `<body>`.

**Различие:** Контейнер действий имеет другого родителя, а контракт хранения данных о состоянии отличается.

**Вероятное влияние:** При использовании аналогичного CSS могут отличаться колоночная компоновка, выравнивание, перенос элементов и адаптивное поведение главного блока.

**Уверенность:** Высокая

**Нужны ли CSS или вычисленные стили?** Да

### 5.4 Карточки профиля

**Структура реальной CRM:** Данные профиля разделены на множество повторяющихся карточек со следующим набором классов:

`crm-card crm-subject-profile-section crm-profile-section crm-form-card crm-section`

Сетки полей дополнительно используют `crm-data-grid`, а отдельные поля — `crm-data-field`.

**Структура Static UIKit:** Страница физического лица использует меньше крупных карточек, специализированные сгруппированные разделы и строки профиля без классов `crm-profile-section`, `crm-data-grid` и `crm-data-field`.

**Различие:** Информация представлена с помощью другой иерархии карточек и другого набора классов.

**Вероятное влияние:** При применении общего CSS или CSS реальной CRM могут отличаться границы карточек, поведение сетки и оформление полей.

**Уверенность:** Высокая

**Нужны ли CSS или вычисленные стили?** Да

### 5.5 Банковские реквизиты

**Структура реальной CRM:** Каждый банковский реквизит представлен стандартной карточкой профиля с трёхколоночной сеткой данных и серверными действиями редактирования и удаления.

**Структура Static UIKit:** Банковские реквизиты используют специализированные элементы `crm-bank-account-card`, семантические поля `<dl>`, бейджи и скрытую форму редактирования, управляемую JavaScript.

**Различие:** Отличаются иерархия компонента, разметка полей и сценарий выполнения действий.

**Вероятное влияние:** Вероятны различия в визуальной группировке и расположении элементов управления. Часть различий может быть результатом намеренного развития прототипа.

**Уверенность:** Высокая

**Нужны ли CSS или вычисленные стили?** Да

### 5.6 Вкладки

**Структура реальной CRM:** Профиль → Банковские реквизиты → Договоры / Счета → Документы → Связи → История. И вкладка профиля, и её панель имеют класс `uk-active`. У секций содержимого есть идентификаторы `bank`, `contracts`, `relations` и `history`.

**Структура Static UIKit:** Профиль → Банковские реквизиты → Документы → Связи → Договоры / Счета → История. Активная вкладка отмечена, но ни одна исходная панель не имеет `uk-active`. Соответствующие идентификаторы секций отсутствуют.

**Различие:** Отличаются индексы вкладок и панелей, исходное активное состояние и цели для ссылок с хешем.

**Вероятное влияние:** UIkit может активировать первую панель во время выполнения JavaScript, однако состояние до инициализации, работа без JavaScript, связывание вкладок и навигация по хешу могут отличаться.

**Уверенность:** Высокая

**Нужны ли CSS или вычисленные стили?** Для подтверждения структурного различия — нет; для оценки итогового отображения — да.

### 5.7 Таблицы

**Структура реальной CRM:** Таблицы договоров и счетов используют `crm-table-wrapper crm-table-card` и внутренний контейнер `crm-table crm-table-compact`. Таблицы связей и истории используют более простую устаревшую структуру: `crm-table-wrapper` непосредственно содержит `table.crm-table`.

**Структура Static UIKit:** Вкладки с данными последовательно используют нормализованную структуру обёрток и специализированные классы таблиц, например `crm-relations-table` и `crm-history-table`.

**Различие:** Для связей и истории отличаются глубина обёрток и набор классов. Также отличаются схемы таблиц:

- Счета: в реальной CRM пять столбцов, в статической странице физического лица — четыре.
- Связи: в реальной CRM пять столбцов, в статической — четыре.
- История: в реальной CRM четыре столбца, в статической — пять.
- Договоры: в реальной CRM есть столбец действий, в статической странице физического лица он отсутствует.

**Вероятное влияние:** Вероятны различия в оформлении таблиц, ширине, выравнивании столбцов, горизонтальной прокрутке и действиях в строках.

**Уверенность:** Высокая

**Нужны ли CSS или вычисленные стили?** Да

### 5.8 Кнопки, ссылки и статусы

**Структура реальной CRM:** Архивация и удаление банковских реквизитов выполняются через POST-формы со скрытыми значениями. В главном блоке есть заблокированная кнопка состояния комплаенса. Закрытие договора выполняется опасной кнопкой. Для статусов используются варианты бейджей `info`, `success` и `warning`, а также специальные элементы истории.

**Структура Static UIKit:** Действия в основном представлены прототипными кнопками или ссылками, управляемыми локальным JavaScript. В главном блоке предлагается действие оформления договора, а изменения истории преимущественно используют обычные бейджи.

**Различие:** Отличаются типы элементов, классы, заблокированные состояния, семантика форм и представленные состояния пользовательских сценариев.

**Вероятное влияние:** Могут отличаться визуальная значимость элементов управления, оформление заблокированных состояний, расположение и поведение. Часть различий связана с разными демонстрационными состояниями, а не с дефектами компонентов.

**Уверенность:** Высокая

**Нужны ли CSS или вычисленные стили?** Да

### 5.9 Формы и модальные окна

**Структура реальной CRM:** Для архивации, удаления банковских реквизитов и закрытия договора используются серверные POST-формы. Закрытие договора выполняется через модальное окно UIkit со скрытыми полями и явными кнопками подтверждения и отмены.

**Структура Static UIKit:** На странице физического лица нет соответствующей формы или модального окна UIkit. Прототипные элементы управления используют локальный JavaScript и альтернативные переходы между страницами.

**Различие:** Отличается архитектура взаимодействия, а не только оформление.

**Вероятное влияние:** Если сценарий закрытия договора должен соответствовать реальной CRM, видимые элементы управления и поведение будут отличаться.

**Уверенность:** Высокая

**Нужны ли CSS или вычисленные стили?** Для подтверждения структурного различия — нет; для сравнения внешнего вида — да.

### 5.10 Документы и пустые состояния

**Структура реальной CRM:** Во вкладке документов отображается карточка «раздел в разработке». Таблица счетов демонстрирует строку `crm-empty-state`.

**Структура Static UIKit:** Вкладка документов полностью заполнена элементами управления и таблицей. В разделе счетов присутствуют демонстрационные строки.

**Различие:** Страницы демонстрируют разные состояния готовности функций и разные состояния данных.

**Вероятное влияние:** Видимое содержимое значительно различается, но это не следует автоматически считать дефектом паритета.

**Уверенность:** Высокая

**Нужны ли CSS или вычисленные стили?** Нет

## 6. Ожидаемые и потенциально проблемные различия

### Ожидаемые различия

- Снимок реальной CRM представляет физическое лицо, а `subject-card.html` — организацию.
- Отличаются имена, идентификаторы, банковские реквизиты, договоры, документы и значения статусов.
- Статический прототип содержит дополнительные сценарии управления документами и счетами.
- Баннер заявки на закрытие на странице организации может зависеть от состояния.
- Доступные действия могут различаться из-за разных состояний комплаенса и договоров.
- Серверные URL и относительные ссылки статических страниц ожидаемо различаются.

### Потенциально проблемные различия

- Действия главного блока используют другую иерархию обёрток.
- Порядок вкладок не соответствует снимку реальной CRM.
- У начальной статической панели отсутствует исходный класс `uk-active`.
- В Static UIKit отсутствуют идентификаторы секций, используемые ссылками реальной CRM с хешем.
- В карточках профиля отсутствуют классы совместимости, систематически используемые в реальном снимке.
- Отличаются схемы таблиц связей, счетов, договоров и истории.
- Таблицы связей и истории используют разные контракты обёрток.
- Если пользовательский сценарий должен соответствовать реальной CRM, отсутствуют модальное окно закрытия договора и семантика его формы.
- Иерархия боковой панели и сопоставление значков неэквивалентны.

## 7. Рекомендуемые следующие шаги

### P0 — уточнить до внесения исправлений

1. Подтвердить, что снимок реальной CRM следует сравнивать прежде всего с `subject-card-individual.html`.
2. Получить снимок реальной карточки организации, если для `subject-card.html` требуется аудит паритета.
3. Получить соответствующие снимки экрана, CSS реальной CRM или вычисленные стили, размеры окна браузера и состояние DOM после выполнения JavaScript.
4. Уточнить, отражает ли заглушка документов текущее состояние продукта или более старую версию.

### P1 — безопасные исправления на основе аудита

1. Согласовать порядок вкладок и панелей.
2. Добавить исходный `uk-active` панели профиля, если это соответствует выбранному способу инициализации UIkit.
3. Добавить отсутствующие идентификаторы секций, используемые ссылками реальной CRM.
4. Добавить классы совместимости `crm-profile-section`, `crm-data-grid` и `crm-data-field` там, где структуры действительно эквивалентны.

### P2 — рискованные исправления, требующие решения

1. Изменить иерархию действий главного блока и соответствующий контракт адаптивного CSS.
2. Согласовать общую иерархию боковой панели и сопоставление значков.
3. Изменить обёртки или схемы таблиц.
4. Заменить прототипные сценарии Static UIKit семантикой серверных форм.
5. Добавить сценарий модального окна закрытия договора.

Эти изменения не следует реализовывать только на основании текущего HTML-снимка.

## 8. Что невозможно проверить только по HTML

- Точные отступы и выравнивание
- Цвета и цветовой контраст
- Семейства, размеры, насыщенность и межстрочные интервалы шрифтов
- Размеры элементов и максимальную ширину
- Границы, скругления и тени
- Адаптивные точки переключения и поведение перекомпоновки
- Поведение переполнения и прокрутки
- Скрытое или видимое состояние после инициализации JavaScript
- Визуальные состояния при наведении, фокусе, активности и блокировке
- Созданные UIkit роли, идентификаторы, ARIA-атрибуты и итоговое состояние панелей
- Особенности отображения в разных браузерах
- Условное присутствие блоков, зависящих от состояния в рабочей системе

## 9. Итоговый вердикт

Между снимком реальной CRM и Static UIKit есть реальные структурные различия. Статическая карточка физического лица является более подходящим объектом сравнения и сохраняет основную систему CRM-компонентов, поэтому текущая реализация не является принципиально неверной или несвязанной с оригиналом.

Некоторые различия, вероятно, ожидаемы из-за разных данных, состояний пользовательских сценариев и уровня готовности функций. Другие различия — прежде всего вложенность главного блока, порядок и состояние вкладок, классы совместимости профиля, схемы таблиц и модальный сценарий — представляют собой вероятные пробелы паритета.

Не следует вносить исправления вслепую, пока не подтверждены целевой тип субъекта и ожидаемое состояние пользовательского сценария. Этот аудит нужно использовать для определения того, какие дополнительные снимки реальной CRM, CSS, скриншоты и состояния браузера необходимо собрать до начала реализации.
