# Frontend-аудит репозитория Vitac86/CRM_Design

Дата аудита: 2026-04-24  
Контекст: дизайн-прототип CRM для передачи внутренней команде разработки под интеграцию с существующей боевой CRM/БД (без превращения проекта в backend-приложение).

---

## 1) Executive summary

Проект в текущем виде хорошо покрывает визуальный прототип и демонстрационные пользовательские сценарии, но **не готов к чистой передаче в интеграцию без архитектурной доработки frontend-слоя**. Главный риск — смешение mock-режима и прикладной логики в компонентах/страницах, из-за чего команда интеграции получит неоднозначную точку входа к API.

Ключевые блокеры передачи:
- нет четкого разделения `demo(mock)` и `api` слоев;
- часть экранов читает данные из `ClientsStore`, часть — напрямую из `src/data/*`;
- отсутствуют единые контракты загрузки/ошибок для будущего асинхронного API;
- не закрыты маршрутизация ошибок/404 и базовая адаптивность shell/layout.

---

## 2) Проблемы по приоритетам

## Critical

### C1. Смешение источников данных (store + прямые импорты mock) => рассинхрон UI
- **Файлы:** `src/components/layout/Topbar.tsx`, `src/pages/TradingPage.tsx`, `src/pages/TradingCardPage.tsx`, `src/pages/DashboardPage.tsx`, `src/components/crm/ReportsPageTemplate.tsx`, `src/pages/DocumentsPage.tsx`, `src/pages/DepositoryPage.tsx`, `src/pages/MiddleOfficePage.tsx` и др.
- **Описание:** в проекте одновременно используются `useClientsStore()` и прямые импорты из `src/data/*`. После изменения данных через store (например, регистрация/архивирование/обновление) часть экранов может показывать старые данные из статичных модулей.
- **Риск для передачи:** разработчики интеграции не смогут предсказать «источник истины» для экрана; высокая вероятность регрессий при подключении API и кэширования.
- **Конкретное исправление:**
  1. Ввести единый data-access слой (`src/features/*/api` + `src/features/*/mock`).
  2. Для всех экранов запретить прямой импорт `src/data/*` из UI-компонентов/страниц.
  3. Оставить mock как **demo provider** (переключаемый режим), но с тем же интерфейсом, что и API-клиент.

### C2. Нет формализованной границы demo/mock и будущего API
- **Файлы:** `src/data/*.ts` (все), `src/app/ClientsStore.tsx`, страницы с прямым импортом данных.
- **Описание:** mock-структуры фактически выполняют роль runtime-хранилища, а не только демо-данных; отсутствуют интерфейсы репозиториев/сервисов для адаптации к backend.
- **Риск для передачи:** при интеграции придется переписывать страницы, а не просто заменить источник данных; существенно растет стоимость встраивания в боевую CRM.
- **Конкретное исправление:**
  1. Определить контракты use-case слоя (`ClientsRepository`, `RequestsRepository`, `ReportsRepository` и т.д.).
  2. Реализовать `Mock*Repository` и `Api*Repository`.
  3. Подключать через DI/флаг режима (`DEMO_MODE=true/false`) на уровне app bootstrap.

### C3. Мутации module-level mock-данных (глобальные побочные эффекты)
- **Файлы:** `src/data/requests.ts`, `src/pages/RequestsPage.tsx`.
- **Описание:** `createRequest()` мутирует экспортированный массив `requests` (`requests.unshift(...)`). Это неявное глобальное состояние, зависящее от порядка навигации и времени жизни сессии.
- **Риск для передачи:** непредсказуемое поведение при интеграции, дубли/рассинхрон между экранами, сложность для тестирования и отладки.
- **Конкретное исправление:**
  1. Перенести операции create/update в единый store/repository.
  2. Сделать `src/data/*` неизменяемыми seed-данными.
  3. Для demo-режима держать runtime-состояние в провайдере (React state/RTK/Zustand), а не в модуле данных.

## High

### H1. Отсутствует fallback маршрутизации (404/error boundary)
- **Файлы:** `src/app/routes.tsx`.
- **Описание:** в роутере нет `errorElement` и `*`-маршрута для Not Found.
- **Риск для передачи:** при встраивании в существующую CRM некорректные URL будут приводить к пустому/неожиданному состоянию без управляемого UX.
- **Конкретное исправление:** добавить `errorElement` на корневой маршрут + `path: '*'` с явной страницей 404/навигацией назад.

### H2. Shell/layout не готов к мобильному и узкому viewport
- **Файлы:** `src/components/layout/AppShell.tsx`, `src/components/layout/Sidebar.tsx`.
- **Описание:** постоянный `fixed` сайдбар шириной `270px` и контент с `ml-[270px]` без mobile-режима (drawer/overlay).
- **Риск для передачи:** при запуске в ограниченных iframe/рабочих окнах CRM макет ломается, часть контента недоступна.
- **Конкретное исправление:** реализовать responsive-shell (бургер + drawer до `lg`, фиксированный sidebar только на desktop).

### H3. Нет стандарта для loading/empty/error состояния API-экрана
- **Файлы:** большинство страниц (`src/pages/*.tsx`), переиспользуемые таблицы.
- **Описание:** сейчас данные синхронны, поэтому отсутствует единый паттерн состояний `loading/error/retry`.
- **Риск для передачи:** после подключения API UX станет неоднородным; команда будет придумывать состояния локально в каждом модуле.
- **Конкретное исправление:** ввести `AsyncState`-паттерн и универсальные компоненты (`PageSkeleton`, `InlineError`, `RetryAction`) + соглашение на уровне страницы.

### H4. Валидация форм фрагментирована и не контрактна
- **Файлы:** `src/pages/ClientRegistrationWizardPage.tsx`, `src/pages/RequestsPage.tsx`, `src/pages/SubjectProfilePage.tsx`, `src/pages/ContractWizardPage.tsx`.
- **Описание:** валидация реализована вручную в компонентах/страницах, без единой схемы полей и маппинга backend-ошибок.
- **Риск для передачи:** высока вероятность расхождения frontend-валидации с backend-правилами и дорогое сопровождение.
- **Конкретное исправление:** вынести схемы (например, Zod/Yup или typed custom validators) и отдельный слой трансформации ошибок API в UI.

## Medium

### M1. Дублирование логики фильтрации/сортировки/пагинации между страницами
- **Файлы:** `src/pages/SubjectsPage.tsx`, `src/pages/RequestsPage.tsx`, `src/pages/TradingPage.tsx`, `src/pages/ArchivesPage.tsx`, `src/pages/CompliancePage.tsx`.
- **Описание:** однотипная логика реализована заново на каждой странице.
- **Риск для передачи:** при изменении требований интеграции правки придется вносить в много мест, растет регрессионный риск.
- **Конкретное исправление:** выделить универсальные hooks (`useTableQueryState`, `useLocalFiltering`, `useSort`) и единый формат query params.

### M2. Непрозрачные доменные константы как строковые литералы UI-уровня
- **Файлы:** `src/data/types.ts`, страницы и утилиты форматирования.
- **Описание:** статусы/типы и служебные значения хранятся как русские строки, тесно связанные с отображением.
- **Риск для передачи:** сложно маппить к backend enum/классификаторам, возможны ошибки локализации и сравнения.
- **Конкретное исправление:** разделить `domain value` и `display label` (внутренние enum-коды + словари отображения).

### M3. Слабая технологическая обвязка качества (нет lint/test/storybook)
- **Файлы:** `package.json`.
- **Описание:** есть только `dev/build/preview`, отсутствуют обязательные quality gates.
- **Риск для передачи:** интеграционная команда не получает автоматических гарантий по регрессиям/стилю/типовым edge-case.
- **Конкретное исправление:** добавить минимум `eslint`, smoke/integration tests (Vitest + RTL), базовые визуальные проверки критичных экранов.

### M4. Тяжелый bundle для прототипа без контроля производительности
- **Файлы:** сборка (`npm run build`), роутинг/страницы.
- **Описание:** единый бандл (без route-level code splitting) вносит риск деградации TTI при встраивании.
- **Риск для передачи:** на слабых рабочих станциях и в корпоративной среде UI будет открываться медленнее ожидаемого.
- **Конкретное исправление:** lazy-load для крупных разделов (`React.lazy` + suspense skeleton), добавить бюджет bundle-size в CI.

## Low

### L1. В репозитории присутствуют артефакты, не относящиеся к runtime-коду
- **Файлы:** корень (`*.png`, `code.html`, `investica_crm_prototype.html`).
- **Описание:** рядом с исходниками лежат дизайн-скриншоты и html-артефакты.
- **Риск для передачи:** шум в репозитории, затрудненный onboarding для команды интеграции.
- **Конкретное исправление:** вынести в `/docs/design-artifacts` или во внешний дизайн-репозиторий.

### L2. Часть заглушек/демо-данных не маркируется единообразно
- **Файлы:** страницы и компоненты, где действия показывают тост «заглушка», а часть просто имитирует сохранение.
- **Описание:** демо-поведение местами явно обозначено, местами — нет.
- **Риск для передачи:** разработчикам сложно быстро понять, что уже «интеграционно готово», а что только UI-макет.
- **Конкретное исправление:** единый `DEMO badge/flag` и комментарии уровня use-case в коде.

## 3) Чек-лист передачи

### ✅ Готово к передаче (как дизайн-прототип)
- Базовая модульная структура React/Vite/TS и единый shell.
- Большой набор экранов CRM-доменов (subjects, compliance, requests, trading, middle/back-office и т.д.).
- Наличие демонстрационных данных и рабочих пользовательских потоков для показа UI.
- Проект собирается production-сборкой.

### ⛔ Нужно доделать до передачи
- Разделить demo/mock и API-слой через интерфейсы репозиториев.
- Убрать прямые импорты `src/data/*` из UI-страниц и перейти на единый data-access.
- Убрать мутации module-level mock-данных.
- Добавить 404/error boundary и единые loading/error/retry состояния.
- Закрыть адаптивность shell для узких экранов/iframe.
- Ввести минимум quality gates (lint + smoke tests).

### 🤝 Нужно обсудить с backend/CRM-командой
- Канонические enum/классификаторы и их кодовые значения (не display-строки).
- Контракты API по разделам (clients, requests, compliance, reports, trading).
- Формат и стратегию пагинации/фильтров/сортировки (server-side vs client-side).
- Политика optimistic updates, конфликтов и блокировок карточек.
- Политика отображения архивных/удаленных сущностей и ошибок валидации от backend.

---

## 4) Рекомендуемый план доработки (коротко)

1. **Спринт 1 (архитектура):** data-access контракты + mock/api провайдеры + перенос чтения данных из страниц в use-case слой.  
2. **Спринт 2 (UX-интеграция):** Async state pattern, 404/error boundary, адаптив shell.  
3. **Спринт 3 (качество):** lint/test CI, bundle budget, документация интеграции (`INTEGRATION_GUIDE.md`).

---

## 5) Примечание о бизнес-логике

Бизнес-логика проекта в рамках аудита **не изменялась**. В отчете описаны только риски и предлагаемые технические исправления, необходимые для безопасной передачи прототипа в интеграцию.
